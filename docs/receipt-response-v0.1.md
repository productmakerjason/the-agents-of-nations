# Submission Receipt Response v0.1

Status: draft  
Project: The Agents of Nations  
Purpose: define the minimum target-system receipt required for an agent to move from `payload_prepared` to `submitted_with_receipt`.

---

## Core rule

A prepared payload is not a completed task.

A submission page is not a receipt.

Agent-generated proof is not completion proof.

The final transition requires target-system proof.

A proof field in an agent output is not proof unless its origin is established.

Self-reported proof can be execution evidence, but not completion proof.

---

## Minimum receipt response

```json
{
  "status": "received",
  "receipt_id": "rcpt_20260527_001",
  "task_id": "agent_discovery_001",
  "received_at": "2026-05-27T00:00:00Z",
  "submission_status": "submitted_with_receipt",
  "proof_origin": "target_system",
  "validation": {
    "status": "passed",
    "required_fields_present": true,
    "task_id_exists": true,
    "schema_version": "0.3-route-hardening"
  }
}
```

---

## Required fields

- `status`
- `receipt_id`
- `task_id`
- `received_at`
- `submission_status`
- `proof_origin`
- `validation`

---

## Field definitions

### `status`

Whether the target system received the submission request.

Allowed values:

```txt
received
rejected
error
```

### `receipt_id`

A receipt identifier generated by the target system.

Agents must not self-generate this as completion proof.

### `task_id`

The submitted task ID.

It should exist in the public task feed.

### `received_at`

The ISO timestamp generated by the target system when the submission was received.

### `submission_status`

The submission state assigned by the target system.

Allowed values:

```txt
submitted_with_receipt
rejected
invalid_payload
```

### `proof_origin`

Where the proof came from.

Allowed values:

```txt
target_system
public_durable_record
agent_self_reported
unknown
```

Accepted as completion proof:

```txt
target_system
public_durable_record
```

Not accepted as completion proof:

```txt
agent_self_reported
unknown
```

### `validation.status`

The target-system validation result.

Allowed values:

```txt
passed
failed
partial
```

---

## Valid terminal states

If a valid receipt exists:

```txt
submitted_with_receipt
```

If no valid receipt exists:

```txt
prepared_not_submitted
submission_attempted_no_receipt
safe_stopped
```

---

## Transition rule

```txt
payload_prepared → submitted_with_receipt
```

requires:

```txt
receipt_id exists
proof_origin = target_system or public_durable_record
validation.status = passed
```

Without these, the agent must not claim completion.

---

## Why this exists

The submit-pressure runs showed that agents could often prepare a valid payload, but refused to claim completion because `/submit` did not return a target-system receipt.

The key distinction is:

```txt
self-reported proof ≠ system-returned proof
```

A payload generated by an agent can be useful execution evidence, but the final completion proof must come from the target system or a durable public record.

---

## v0.1 non-goals

This version does not define:

- payment settlement
- reputation scoring
- full identity verification
- fraud adjudication
- dashboard workflows
- marketplace listing logic
- long-term storage guarantees

The only goal is to test whether agents can distinguish:

```txt
payload_prepared
```

from:

```txt
submitted_with_receipt
```

using a target-system receipt.
